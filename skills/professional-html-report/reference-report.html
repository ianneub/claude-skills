The Bottom Line

The environment is production-stable but carries three critical exposures and roughly 31% of monthly spend tied up in idle or oversized resources. None are hard to fix: the two highest risks — a publicly shared RDS snapshot and unrotated root access keys — can be closed this week. Reliability is genuinely solid at 99.94% uptime, but a single-AZ production database undercuts the whole posture. Fix the data-layer items first; the cost recovery (~$11.4k/mo) funds the remediation several times over.

04 Blast radius

Blast Radius

The public RDS snapshot (finding #1) contains a full copy of the customer-orders database, including PII, and has been shareable since the last manual restore 38 days ago. Treat it as a potential disclosure: revoke sharing, rotate database credentials, and review CloudTrail for CopyDBSnapshot / RestoreDBSnapshotFromS3 calls by unknown principals before doing anything else.

05 Action items

  1. Revoke the public RDS snapshot and audit CloudTrail for external copies — today.
  2. Rotate then delete the root access keys; move automation onto scoped IAM roles — this week.
  3. Convert the production database to Multi-AZ and extend backups to 35 days — this week.
  4. Enable S3 default encryption org-wide via SCP and close the two internet-facing security groups.
  5. Right-size the nine idle/oversized instances and release the 14 unattached volumes — ~$11.4k/mo recovered, reviewed monthly.
