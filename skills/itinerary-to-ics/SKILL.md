# Itinerary to ICS

Turn a travel itinerary (flights, hotels, meetings) into an `.ics` file whose
locations show as **tappable maps** in Apple Calendar.

## The core problem

Apple Calendar **ignores plain `LOCATION:` text** when deciding whether to draw a
map. A map pin appears only when the event has an `X-APPLE-STRUCTURED-LOCATION`
property ending in a `geo:LAT,LON` URI. The richest version also carries an
`X-APPLE-MAPKIT-HANDLE` тАФ an opaque Apple POI token.

Two hard constraints:

1. **You cannot fabricate a valid `X-APPLE-MAPKIT-HANDLE` offline.** It is an
   encoded MapKit place id generated by Apple's servers.
2. **A `geo:` URI + `X-TITLE` alone is enough for a working pin** (confirmed:
   many real Apple events have no handle). The handle only upgrades the pin to a
   rich POI card.

So there are two ways to get a location block:

- **Best (rich POI):** harvest the exact block from a calendar export (`.ics`)
  that already contains that place тАФ handle and all. Reuse it verbatim.
- **Good (plain pin):** supply latitude/longitude yourself; emit a handle-less
  structured location. Look up coordinates once (drop a pin in Apple Maps, or any
  geocoder).

## Format rules Apple's parser is strict about

These were the actual failure causes when maps didn't show:

- **Fold every line to тЙд 75 octets.** Continuation lines start with a single
  space. An unfolded 200-char `X-APPLE-STRUCTURED-LOCATION` is silently dropped.
- **Use CRLF (`\r\n`) line endings**, not LF.
- **`X-TITLE` is NOT quoted** even when it contains spaces/apostrophes.
- **`X-ADDRESS` IS quoted** and uses real commas, not `\n`.
- The provided scripts handle all of this. If hand-writing, replicate exactly.

## Workflow

A curated airport `library.json` **ships with this skill** тАФ `make_ics.py` reads
it by default, so for flights between major US airports you can skip straight to
writing the itinerary. Building/extending the library is only needed for a place
it doesn't cover yet (see "Resolving an airport not in the library").

### 1. Write the itinerary JSON

Copy `itinerary.example.json` and fill in the trip. Each event names a `place`
that is matched (case-insensitive substring) against the library. For a place not
in the library, give coordinates inline instead (see template).

### 2. Generate the ICS

```
python3 make_ics.py mytrip.json ~/Downloads/mytrip.ics
```

It looks up each place, reuses the exact structured location, pulls only the
needed `VTIMEZONE` blocks, folds to 75 octets, and writes CRLF. It prints a
warning for any place it couldn't find (those events get no map) тАФ resolve those
with the section below.

## Resolving an airport not in the library

When `make_ics.py` prints `WARNING: place not found`, or you know an airport
isn't there yet, pick the highest option you can:

1. **Best тАФ harvest a real handle.** If any calendar export contains that
   airport, run `build_library.py export.ics`. You get the rich POI card, and it
   persists for every future trip. Worth asking the user whether they have an
   export (or a past trip on their calendar) that flew through it.

2. **Good & reusable тАФ add it by coordinates.** No export? Add a handle-less
   pin that persists in the library:

   ```
   python3 add_place.py "Boise Airport" "43.564386,-116.222861" \
       "3201 W Airport Way, Boise, ID 83705, United States"
   ```

   The map pin works; it just isn't a rich POI card. Next time an export with the
   real place shows up, `build_library.py` upgrades this entry automatically.

3. **One-off тАФ inline coordinates.** For a single use, skip the library and put
   `place_title` + `address` + `geo` directly on the event (see JSON shape).

**Getting coordinates** (decimal degrees, **latitude first**, e.g.
`43.564386,-116.222861`):
- For well-known commercial airports, the published lat/long is easy to look up
  (search the airport name or its IATA/ICAO code).
- Google Maps: right-click the spot тАФ the top of the menu is `lat, lon`; click to
  copy.
- Apple Maps: search/drop a pin, Share тАФ the URL contains `ll=lat,lon`.
- Wikipedia: most airports list "Coordinates" in the infobox.

Confirm the coordinates land on the terminal, not a random field, before adding.

## Bundled files

- `library.json` тАФ the shipped location library: curated airports (with Apple
  MapKit handles) plus US `VTIMEZONE` definitions. Safe to distribute тАФ airports
  only, no personal locations. All three scripts default to this file.
- `make_ics.py` тАФ itinerary JSON тЖТ folded, CRLF, map-ready `.ics`.
- `add_place.py` тАФ add an airport to `library.json` by coordinates (handle-less
  pin). Won't overwrite an entry that already has a MapKit handle.
- `build_library.py` тАФ harvest places + timezones from an Apple `.ics` export
  into `library.json`. **Harvests every place, including personal ones тАФ curate
  down to airports and review the diff before committing.**
- `itinerary.example.json` тАФ template showing library lookup and the
  coordinate-only fallback.

## Itinerary JSON shape

```json
{
  "prodid": "-//Me//Itinerary//EN",
  "dtstamp": "20260101T000000Z",
  "events": [
    {
      "uid": "unique-stable-id@me",
      "summary": "Delta 3057: CHA -> ATL",
      "description": "line one\\nline two\\n\\nFlightAware: https://...",
      "url": "https://flightaware.com/live/flight/DAL3057",
      "place": "Chattanooga Metropolitan Airport",
      "start": {"tz": "America/New_York", "dt": "20260609T160500"},
      "end":   {"tz": "America/New_York", "dt": "20260609T170400"}
    }
  ]
}
```

Notes:
- `dt` is local wall-clock time `YYYYMMDDThhmmss`; `tz` is an IANA TZID that must
  exist in the library (the shipped library includes the common US zones;
  `build_library.py` adds more from an export).
- Use `\\n` inside `description` for line breaks.
- Coordinate-only fallback for an event: replace `place` with
  `"place_title"`, `"address"`, and `"geo": "LAT,LON"`.

## Domain knowledge

- **Time zones matter for flights.** Each leg's depart/arrive times are *local*
  to its airport. Set `start.tz` to the departure airport's zone and `end.tz` to
  the arrival airport's zone so durations render correctly across zones.
- The event's location should be the **departure** point (where the traveler
  needs to be) unless the user says otherwise.
- US airport timezones: most Eastern airports тЖТ `America/New_York`; Detroit тЖТ
  `America/Detroit`; Mountain тЖТ `America/Denver`; Arizona (no DST) тЖТ
  `America/Phoenix`; Central тЖТ `America/Chicago`; Pacific тЖТ
  `America/Los_Angeles`.

## Re-import gotcha

Apple Calendar keys events by `UID`. Re-importing a file with an existing UID
often **keeps the old location and won't re-geocode**. If a map doesn't appear
after an update, tell the user to **delete the old events first**, then import.

## Common mistakes

| Mistake | Fix |
|---------|-----|
| Plain `LOCATION:` only, expecting a map | Add `X-APPLE-STRUCTURED-LOCATION` with a `geo:` URI. |
| Unfolded long structured-location line | Fold to тЙд75 octets; Apple drops overlong lines. |
| LF line endings | Use CRLF. |
| Quoting `X-TITLE` / putting `\n` in `X-ADDRESS` | `X-TITLE` unquoted; `X-ADDRESS` quoted with real commas. |
| Fabricating a MapKit handle | Can't be done offline тАФ harvest from an export, or go handle-less with coordinates. |
| Committing personal places into the shipped `library.json` | The bundled library is airports only. `build_library.py` harvests everything from an export тАФ curate to airports and review the diff before committing. |
| Same-UID re-import shows no map | Delete the old events first, then re-import. |
| Same `tz` for all legs of a multi-zone trip | Use each airport's local TZID for depart/arrive. |
| Airport missing from the library | Resolve it (see "Resolving an airport not in the library"): harvest an export, `add_place.py` by coordinates, or inline `geo`. |
